## 1. 产品定位与设计原则

### 1.1 产品定位

> **Admin Web 是 Smivo 的"运营自治工具",不是"程序员的远程控制器"**。
> 好的 Admin 让运营越来越独立(改字典、调 Flag、维护词库都不需要程序员);差的 Admin 永远依赖发版。

### 1.2 设计原则

- **任务导向**:每个页面回答一个明确的运营问题,而不是堆砌数据
- **信息密度优先**:运营人员每天看 8 小时,不追求 Smivo App 的圆角玻璃拟物风
- **危险操作必须二次确认**:封禁、强制下架、删除都要走确认对话框
- **理由必填**:任何对用户产生负面影响的操作必须有详细理由
- **审计先行**:所有写操作自动留痕到 admin_audit_logs
- **多校隔离**:每个页面的数据范围由顶栏学校切换器决定

### 1.3 视觉语言

| 维度 | 要求 |
|---|---|
| 配色 | 浅灰底 + 深色文字 + 语义色;禁止品牌渐变 |
| 字体 | 等宽数字字体;无衬线标题 |
| 密度 | 信息密度高,不留大块留白 |
| 暗黑模式 | 必须支持 |
| 动效 | 极少,仅状态变化的瞬间反馈 |
| 响应式 | 优先 PC,移动端能看即可 |

## 4. 整体框架(Shell)

### 4.1 顶栏(TopBar)

```
┌─────────────────────────────────────────────────────────────────────┐
│ Smivo Admin > 内容审核 > Listing 审核                                 │
│ [🏫 当前: Smith ▼] [🔍 全局搜索]  [🔔 5]  [👤 Alice ▼]              │
└─────────────────────────────────────────────────────────────────────┘
```

**组成**:
- 左:Logo + 面包屑(基于当前路由动态生成)
- 中:学校切换器(详见 §4.3) + 全局搜索框
- 右:告警铃铛(SLA 超时、严重举报涌入) + 当前 admin 头像下拉

**全局搜索行为**:
- 同时搜:用户(邮箱/昵称)、Listing(标题/ID)、订单(ID)
- 搜索结果按类型分组展示
- 范围跟随当前学校切换器(全平台视图时跨校搜)

**告警铃铛**:
- 数字 = 当前 admin 需要关注的高优事件数
- 包括:即将超时的草稿、SLA 超时未处理的 Listing、严重举报涌入
- 点击展开下拉清单,每条带跳转链接

### 4.3 学校切换器

```
[🏫 Smith College ▼]
  ↓ 点击展开
  ┌────────────────────────────┐
  │ 🌐 全平台视图(汇总数据)    │  ← 仅平台超管 / 全平台权限审核员可见
  ├────────────────────────────┤
  │ 🏫 Smith College ✓         │  ← 当前选中
  │ 🏫 Mt Holyoke              │
  │ 🏫 Amherst                 │
  └────────────────────────────┘
```

**行为定义**:
- **平台超管**:显示所有学校 + "全平台视图"
- **平台审核员**:仅显示已授权的学校(若被授权全平台也显示"全平台视图")
- **学校管理员**:不显示切换器,顶栏直接显示固定的学校标识

**默认逻辑**:
- 跨校管理员**默认进入单校视图**(读取 localStorage 中"上次选的学校")
- 首次登录(无 localStorage):进入第一个授权学校

**切换效果**:整个 Admin Web 内容区**所有数据自动按当前学校过滤**。Zustand 全局状态 `currentSchoolScope` 触发所有页面 React Query 重新拉取。

### 6.5 ★ 购物车工作流(核心特性)

#### 数据库设计

```sql
CREATE TABLE moderation_drafts (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  admin_id        uuid NOT NULL REFERENCES user_profiles(id),
  target_type     text NOT NULL CHECK (target_type IN ('listing','chat_report','user_report','feedback')),
  target_id       uuid NOT NULL,
  college_id      uuid NOT NULL REFERENCES colleges(id),  -- 多校隔离
  
  decision        text NOT NULL CHECK (decision IN ('approve','reject','takedown','warn','ban')),
  rule_violated   text,
  reason_detail   text,
  
  created_at      timestamptz DEFAULT now(),
  updated_at      timestamptz DEFAULT now(),
  
  UNIQUE (target_type, target_id)  -- 同一目标只能有一个草稿
);
CREATE INDEX idx_drafts_admin_college ON moderation_drafts(admin_id, college_id, created_at DESC);
```

#### 工作流

```
列表卡片 [✅ 通过]
        ↓
    草稿表 INSERT decision='approve'
    (无对话框,1 秒处理)
        ↓
    卡片状态变化:浅灰底色 + 绿色"已标记通过"
    顶部"💼 待提交队列"数字 +1

列表卡片 [❌ 拒绝]
        ↓
    弹出"拒绝理由对话框"(强制必填)
        ↓
    草稿表 INSERT decision='reject' + reason_detail
        ↓
    卡片状态变化:浅灰底色 + 红色"已标记拒绝" + 显示理由
        
顶栏 [💼 我的待提交队列(5)] → 点击打开抽屉
        ↓
    抽屉显示所有草稿,可单条撤销/改理由
        ↓
    [🚀 确认提交全部 5 条]
        ↓
    Edge Function 逐条处理,实时进度反馈
        ↓
    成功:数据生效 + 通知用户 + 写审计日志 + 草稿删除
    失败:草稿保留,允许重试
```

### 6.6 ★ 五层防忘机制

防止 admin 草稿堆积忘提交:

| 层级 | 触发条件 | 表现 |
|---|---|---|
| L1 视觉 | 草稿数 > 0 | 顶部"💼 我的待提交队列"按钮呼吸高亮(橙色) |
| L2 警告 | 任意草稿剩余 < 1h | 顶部红色横幅:"⚠️ 你有 1 条草稿即将超时" |
| L3 推送 | 草稿超过 30 分钟未提交 | OneSignal 推送到 admin 手机 |
| L4 退出拦截 | 关浏览器/退出登录 | 强制弹窗:"您有 5 条草稿未提交,确认退出?" |
| L5 工作台告警 | 进入 Dashboard | 顶部独立告警条:"⚠️ 您有 3 条草稿,1 条即将超时" |

**超时草稿不自动提交**——保护 admin 判断权。Listing 多压几小时由系统接受。

### 6.7 协作机制

- **Realtime 同步**:订阅 `moderation_drafts` 表变化,其他 admin 标记后实时刷新到当前页面
- **占用展示**:被其他 admin 标记的卡片显示"👤 Bob 已标记为通过(待 Bob 提交)",当前 admin 不能操作
- **冲突避免**:UNIQUE 约束保证同一 target 只有一条草稿

### 6.8 批量操作

- 卡片左侧勾选框 → 选中 N 条
- 顶部出现批量按钮:[批量通过] [批量拒绝]
- 批量拒绝时弹出对话框,**填一个共同理由**应用到所有选中
- 批量操作仍走草稿(进入草稿队列)

### 6.9 自动刷新

- 每 60 秒自动 GET 一次最新数据
- Realtime 推送变化也会触发刷新
- 手动刷新按钮始终可用

### 6.10 数据接口

- `GET /admin/moderation/listings?priority=&status=&page=&college_id=`
- `POST /admin/moderation/drafts` — 加入草稿 `{target_type, target_id, decision, ...}`
- `PATCH /admin/moderation/drafts/:id` — 改判定/理由
- `DELETE /admin/moderation/drafts/:id` — 撤销
- `POST /admin/moderation/drafts/submit` — 批量提交 `{draft_ids: [...]}`
- `GET /admin/moderation/drafts/mine` — 我的草稿列表

```sql
  scheduled_at    timestamptz,                       -- NULL=立即
  
  -- 状态
  status          text DEFAULT 'draft' CHECK (status IN ('draft','scheduled','sending','sent','failed','cancelled')),
  
  -- 统计
  recipients_count int,
  delivered_count  int DEFAULT 0,
  opened_count     int DEFAULT 0,
  clicked_count    int DEFAULT 0,
  failure_breakdown jsonb,
  
  -- 元数据
  onesignal_id     text,
  created_by       uuid REFERENCES admin_users(user_id),
  created_at       timestamptz DEFAULT now(),
  sent_at          timestamptz
);

CREATE TABLE push_templates (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name            text NOT NULL,
  title_template  text NOT NULL,
  body_template   text NOT NULL,
  default_deep_link text,
  created_by      uuid REFERENCES admin_users(user_id),
  created_at      timestamptz DEFAULT now()
);
```

## 26. 已知风险与待办

| 风险 | 缓解策略 | 状态 |
|---|---|---|
| Service Role Key 泄漏 | 严格只在 Edge Function | 设计已规避 |
| Admin 权限被盗用 | 强制审计日志 + 重要操作二次确认 | 设计已覆盖 |
| 敏感词被绕过(谐音变体) | MVP 不解决,未来接 OpenAI Moderation | 已记录 |
| 误封用户引发申诉 | 临时封禁优先 + 必填详细理由 | 设计已覆盖 |
| 草稿超时 Listing 多压几小时 | 接受此风险,五层防忘机制保障 admin 不忘 | 已知风险 |
| 多 admin 协作冲突 | UNIQUE 约束 + Realtime 同步 + 占用展示 | 设计已覆盖 |
| 跨校 admin 误操作 | 顶栏学校切换器 + 审计日志记录目标学校 | 设计已覆盖 |
| 推送被滥用打扰用户 | 受众预估数 + 模板复用 + 审计日志 | 部分缓解 |
| 反馈被刷分滥用 | 频率限制(5/天)+ admin 判定才发分 | 设计已覆盖 |
| 图片违规无法自动检测 | MVP 仅人工审,5 起后接 AI | 已记录 |
| 数据字典误删导致历史数据孤儿 | 只能停用不能删除 + 系统级保护 | 设计已覆盖 |
| 直接登 Supabase 控制台改数据绕过审计 | 纪律约束 MVP,未来上数据库触发器双保险 | 已记录 |
